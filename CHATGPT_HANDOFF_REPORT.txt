4. Important Code Observations

- app.py is still very large and acts as a service container, router, and runtime bootstrap in one file.
- services/order_transaction_service.py is more mature than the old order flow, but create_order(...) is still less transactionally disciplined than dispatch/delivery/cancel:
  - it does inventory mutation
  - agreement generation
  - agreement append
  - queue side effects
  - but not under the same explicit transaction journaling/rollback pattern used by dispatch/delivery/procurement
- services/procurement_transaction_service.py emits its own events by writing files directly through _emit_event(...), while order flow uses services/event_dispatcher.py. Event architecture is inconsistent.
- Transaction ID generation is local to each coordinator:
  - procurement_transaction_service has its own counter
  - order_transaction_service has its own counter
  - both generate TXN-<year>-<counter>
  - this can produce duplicate transaction IDs across subsystems
- Agreement IDs and order IDs are not yet globally standardized to a single shared allocator.
- DomainWriteGuard, InventoryDomainService, AgreementDomainService, and storage adapters are now present in app.py, but the runtime still mixes direct service calls and domain abstractions. The layering is not fully settled yet.
- Health recovery is manual from the dashboard. There is no startup-wide unified transaction recovery pass across all coordinators.
