Architecture Rules

The storage philosophy is the architecture.

What Must Exist

  • JSON files for structured records
  • CSV for reports
  • TXT logs for operational traces
  • PDF and image proofs inside Drive
  • Manufacturer-specific folders with shared and private zones

What Must Not Exist

  • PostgreSQL
  • MySQL
  • MongoDB
  • Firebase
  • Any new persistence layer that breaks data sovereignty
Federated Storage

Each manufacturer owns their own Google Drive container.

/MANDITRADE_MANU101/ ├── shared_zone/ │ ├── inventory.json │ ├── procurement.json │ ├── agreements.json │ ├── wallet_summary.json │ └── subscription.json └── private_zone/ ├── clients.json ├── api_keys.json ├── invoices/ ├── payments/ └── business_reports.csv

Ownership Model

  • Each manufacturer controls their own root folder
  • Shared zone supports runtime collaboration and limited governance visibility
  • Private zone is restricted to the manufacturer and authenticated runtime session
  • Admin access stops at the privacy boundary
Access Boundaries

Shared zone and private zone are deliberately unequal.

Shared Zone

Runtime-visible governance layer

  • Contains: inventory, procurement, agreements, subscription, wallet summary
  • Accessible by: application runtime and limited admin governance logic
  • Purpose: exchange, pricing enforcement, agreement orchestration
Private Zone

Manufacturer-only operational layer

  • Contains: clients, API keys, invoices, payments, internal reports
  • Accessible by: manufacturer and authenticated runtime session only
  • Rule: admin must never access this zone
User Hierarchy

Admin governs the network, manufacturers operate it, clients transact within it.

Admin

  • Product registry
  • Mandi pricing and MRP governance
  • Subscriptions and feature flags
  • Procurement governance
  • Manufacturer onboarding

Cannot: view clients, invoices, API keys, or any private zone.

Manufacturer

  • Manage inventory and clients
  • Manage orders and APIs
  • Join procurement exchange
  • Manage agreements

Cannot: override MRP or mandi price, or access other manufacturers.

Client

  • Self-onboard
  • Browse products
  • Place orders
  • Track invoices and agreements

Each client belongs to exactly one manufacturer.

Runtime Security

OAuth access is temporary, scoped, and destroyed after session use.

Manufacturer login → Google OAuth consent → Refresh token fetched securely → Temporary access token generated at runtime → Drive accessed dynamically → Shared/private files resolved per role → Session logout → Session token destroyed

Security Requirements

  • Role-based access control
  • Runtime session isolation
  • Drive permission validation
  • Private-zone protection
  • Encrypted token storage

Never expose: raw OAuth secrets, Drive folder IDs publicly, raw links, or globally cached private data.

Procurement Exchange

If inventory is insufficient, the network becomes the sourcing engine.

Exchange Flow

  1. Manufacturer creates procurement request
  2. Request is published to shared zone
  3. Other manufacturers evaluate availability
  4. One manufacturer accepts the request
  5. Agreement is auto-generated
  6. 50% advance is collected before confirmation

Agreement Lifecycle

DRAFT ACCEPTED ADVANCE_PENDING CONFIRMED DISPATCHED CLOSED

Every procurement order produces both agreement JSON metadata and a PDF agreement document.

Pricing Governance

Prices are centrally governed; execution is decentralized.

Pricing Rules

  • Admin sets fixed mandi price
  • Admin sets fixed MRP
  • Manufacturers cannot override governed prices
  • Commission logic stays policy-driven, not manually negotiated per order
MRP = Rs 145 Mandi Price = Rs 120 Net Profit = Rs 25 Admin Share = 50% of net profit = Rs 12.5 Manufacturer Share = Rs 12.5
